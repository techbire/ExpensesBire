# AI Usage Log

This document details the AI tools used during the development of the Shared Expenses application, the key prompts utilized, specific incorrect outputs generated by AI models, how they were detected, and how the codebase was changed to prevent them.

## 1. AI Tools Used
- **Gemini 3.5 Flash (High)**: Main assistant for planning, code implementation, debugging, test generation, and documentation drafting.

---

## 2. Key Prompts

### For Database Architecture
> "Design a relational schema using PostgreSQL and Prisma ORM for a shared expenses application. The schema must support users, groups, group memberships with join/leave dates, manual expenses, settlements, an import review workflow with quarantined import rows and anomalies, audit logs, and alias mappings."

### For Anomaly Rules Engine
> "Write a TypeScript function `detectAnomalies(row: RawExpenseRow, groupBaseCurrency: string)` that checks a raw CSV row for the following: missing payer, missing currency, commas in amount fields, zero value rows, negative amount rows, foreign currencies, keywords that indicate settlements disguised as expenses, and split percentage mismatches."

### For Split Calculation Engine
> "Implement calculation rules for EQUAL, UNEQUAL, PERCENTAGE, and SHARE splits using high-precision math (Decimal.js). Ensure that any remainder/rounding difference is deterministically balanced against the first participant's share so that the sum of split amounts equals the base expense amount."

## 3. Incorrect AI Outputs, Detections, and Changes

### Mistake 1: Using Javascript Floating Point Math
- **Incorrect Output**: The initial implementation used standard JavaScript native numbers (`number`) for expense calculations and percentage splits:
  ```typescript
  const baseShare = totalAmount / members.length;
  // resulted in precision errors like: 33.333333333333336
  ```
- **How It Was Detected**: During development verification, a split of a `100 INR` expense among 3 members summed to `100.00000000000001` or `99.99999999999999` instead of exactly `100 INR`. This violated the relational database constraint and split validation checks.
- **What Changed**: The engine was completely refactored to use `Decimal.js` for all monetary variables and splits (with a high-precision setting of 20 decimal places, rounding to 4 decimal places). A remainder adjustment logic was added to add the difference to the first person's share:
  ```typescript
  const baseShare = totalAmountBase.dividedBy(count).toDecimalPlaces(4);
  const sumBase = baseShare.times(count);
  const difference = totalAmountBase.minus(sumBase);
  const adjustedShare = index === 0 ? baseShare.plus(difference) : baseShare;
  ```

### Mistake 2: Ambiguous Date Parsing Assumption
- **Incorrect Output**: The AI drafted a parser that assumed standard ISO or US format (`MM-DD-YYYY`) date parsing natively using `new Date(row.date)`.
- **How It Was Detected**: In the provided `expenses_export.csv`, rows like `04-05-2026` (which represents 5th April in DD-MM-YYYY format, used locally) were parsed as May 4th, 2026. This caused a membership violation because Meera moved out on Sunday (end of March), making her active on April 5th but inactive on May 4th.
- **What Changed**: Standardized on checking for date ambiguity. Standard format was enforced as `DD-MM-YYYY`. Any date format that is ambiguous or deviates from the standard pattern (e.g. `Mar-14` or slash/dash swaps) is flagged as an `AMBIGUOUS_DATE` anomaly and requires the user to explicitly confirm/resolve it in the UI before committing.

### Mistake 3: Literal String Match for "paid back" Settlements
- **Incorrect Output**: In `anomalyEngine.ts`, the settlement keyword check used a literal substring comparison:
  ```typescript
  if (descLower.includes("paid back")) { ... }
  ```
- **How It Was Detected**: The unit tests failed on the CSV row description "Rohan paid Aisha back". Because "Aisha" was positioned between "paid" and "back", the literal check `.includes("paid back")` returned `false`, and the settlement was not flagged by description.
- **What Changed**: Modified the condition to look for the presence of both "paid" and "back" in the description, or specific keywords, ensuring it catches structured sentences like "paid [Name] back" or "paid back [Name]":
  ```typescript
  if (descLower.includes("paid back") || (descLower.includes("paid") && descLower.includes("back")) || descLower.includes("settle") || descLower.includes("deposit share")) { ... }
  ```
